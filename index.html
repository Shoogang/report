Section 01 / 项目背景

从"搭建型平台"
到"使用心智"

在迭代 4.0 之前，ADP 是一个标准的"搭建型"开发平台——而这次迭代要让"使用"心智长出来。

1.1 ADP 的过去：一个标准的"搭建型"平台

图位 01-01
ADP 旧版主界面截图
以「应用开发」为主的低代码搭建形态 · 待替换为真实截图

腾讯云智能体开发平台（ADP）原本的形态，是一个面向开发者的智能体搭建与开发平台：

  • 它的核心场景是「搭建 → 发布」的开发链路
  • 它的产品定位偏向"开发者搭建好智能体，再把它发布给 C 端客户使用"
  • 它的整体心智是 "这是一个工具，用来产出智能体"

1.2 上一版本的转折：「智能工作台」首次登场

图位 01-02
智能工作台首次上线时的对话页面截图
标注"对话入口"作为新心智的萌芽 · 待替换

直到上一个版本，平台菜单里多了一个新成员——「智能工作台」。它是一个对话页面，让平台不再只有"低代码搭建/管理"这一种打开方式，多了一种"直接对话使用"的能力。但它当时还只是一个入口，还没真正撑起"使用"这件事的完整心智。

1.3 4.0 的命题：让"使用"心智长出来

4.0 不只是功能堆叠，而是一次"产品灵魂"的重新校准。
本次围绕智能工作台做了大量功能迭代，方向只有一个——让"使用"成为 ADP 与"搭建"并列的另一条主轴。

这意味着，4.0 要回答两个根本问题：

  • 使用心智从哪里长出来？——产品形态上要有承载使用的"家"
  • 使用心智如何与原本的搭建心智共存？——结构上不能割裂，也不能混乱

Section 03 / 使用断点

故事开始
一个被忽视的"使用断点"

在产品需求被提出之前，先让我们直面问题——ADP 这个平台，"使用"这件事其实从未被沉淀过。

3.1 痛点 ① · 员工用智能体，只能靠一条"复制链接"

图位 03-01
现状流程示意图
开发者搭建完应用 → 复制体验链接 → 员工点开使用
"复制链接"这一步是产品化断点

开发者在 ADP 里搭建完一个应用，能很顺畅地把它发布给 C 端——微信客服、企微、各种终端渠道，链路清晰、有产品化的承载。

但如果这个企业，想把智能体给自己的员工使用呢？（比如一个员工提效类的金融助手、客服助手、数据分析助手）答案有点尴尬：

PAIN POINT 01
目前唯一的方式，是复制一条体验链接，让员工点开使用
  • 它没有沉淀为产品内的功能入口
  • 它没有"我的应用"这个概念
  • 它没有"家"
这背后反映的是 ADP 原本的产品定位偏差——
平台原本更偏向"搭建完给 C 端客户用"，对"给企业自己的员工用"这个场景，是缺位的。

3.2 痛点 ② · 应用广场是"广场"，不是"工位"

图位 03-02
应用广场截图
行业精选 / 全部应用 / 各种官方精品应用陈列
每次使用都要重新进入广场 = 体验断点

ADP 还有一个「应用广场」，里面有大量官方精品应用，按行业分类陈列——金融、医疗、零售……员工可以直接点进去，跟某个应用对话。但广场的本质是"逛"，不是"用"：

PAIN POINT 02
广场是"逛"的地方，不是"用"的地方
  • 员工每次想用某个应用，都要重新进广场、重新点开
  • 没有"我常用的应用"这件事
  • 没有任何"使用沉淀"的能力

3.3 产品诉求 → 设计命题

两个痛点合起来看，问题就更清晰了：

ADP 平台里，"使用"这件事是发生过的，但从未被沉淀过。

基于上述背景，产品给出了明确诉求：

沉淀一种"应用 → 对话使用"的产品形态，助力员工工作提效。

同时，借这次 4.0 的契机，产品还有一个更高维度的目标：

不仅要凸显 ADP「能搭建」的心智，更要立起「能直接用」的心智。

—— 这就是这次设计推演的起点。

A
✕ 退场

输入框 @ 应用
最轻、最灵活，但生不逢时

图位 04-A1 + 04-A2
输入框 @ 应用方案 + 当前 Skills 调用类比图
高亮 @ 触发的应用列表 popover · 标注交互方式同源

在智能工作台的输入框里，支持 @ 应用——应用来自「企业应用开发」或「应用广场」，整体逻辑类似今天调用 Skills 的方式。

这个方案的优势
  • 干净、轻、灵活度极高：不需要在产品结构上动任何骨架
  • 认知模型清晰：用户能完全理解这是一种"调用专家 Agent"的能力
  • 应用作为自然语言的一部分：可以被组合、被引用，而不是孤立功能
  • 多专家协同：用户可以同时调用多个应用专家一起解决一个问题，灵活度天花板很高
为什么这么好的方案没有被采纳
维度阻断点
技术现有架构无法支持，赶不上发布节奏
产品定位产品想要的不是"调用"，而是"直接用"——希望"使用"心智更前置、更显性
方案 A 退场。但它埋下了一个关键伏笔——
如果不是"调用"逻辑，那"应用"应该住在产品的哪里？
B
★ 我的第一推荐

把"应用"放进智能工作台
结构最干净的推荐方案

图位 04-B1 + 04-B2
智能工作台左侧任务栏中嵌入"应用 / 远程终端 / 定时任务"
标注三类入口的位置 · hover / 展开收起状态
设计推理 · 一步步来
  • Step 1：既然产品要的是"直接用"，那"应用"的本质就是"使用"。它应该住在承担"使用"心智的地方。
  • Step 2：扫一眼整个 ADP 平台菜单——除了「智能工作台」，下面所有模块（智能体管理、资源中心、平台管理）全都是搭建/管理链路，是给开发者的。
  • Step 3：所以平台里真正承担「使用」定位的，只有智能工作台一个地方。
    → 结论：把"应用"作为智能工作台内部的功能，是最合逻辑的位置。

🌟 惊喜的发现：3 个需求其实是结构同构的

正在这个时候，我手上还有另外两个需求——支持定时任务、支持远程终端渠道。我把它们和"应用"摆在一起观察，发现它们在交互结构上高度一致：

类型 中间层（具体内容） 末端（细粒度任务）
应用 金融助手 任务 1：问 A 股 · 任务 2：问美股
远程终端 企微渠道 任务 1：某个单聊 · 任务 2：某个群聊
定时任务 定时学英语 按定时频率，每隔一段触发一次任务
我把它们抽象为同一种结构——「类型 → 具体内容 → 任务列表」，统一在左侧任务栏里支持展开/收起。一套结构覆盖三种需求，避免三个功能各做各的、造成产品体验的碎片化。

🤔 但这个方案带出了 2 个待决策的问题

问题 1 · 这些功能入口是否应该固定显示？
图位 04-B4
入口固定与否的方案示意
对比 "固定入口" vs "非固定入口" 两种状态

应用、定时任务、远程终端——它们是常驻菜单，还是用户用到时才出现？

问题 2 · 应用任务、定时任务，要不要进入"最近任务"列表？
图位 04-B5
"最近任务"归属问题方案示意
对比"全部混排"vs"重命名 + 应用任务独立"两种思路

这里有一个技术约束需要考虑：

  • 「新建任务」走的是通用 Agent
  • 「具体应用」是配置好模型 / Skills 的独立 Agent
  • 技术上不希望从"新建任务"里能直接创建应用类任务

→ 所以这里有两种解法：

  • 解法 a：把应用任务、定时任务都进"最近任务"，用户感知统一
  • 解法 b：把"最近任务"重命名为"普通任务"，让用户理解"新建任务"只能新建普通任务，应用任务独立列表

⚔️ 然后，挑战来了：大老板的根本质疑

💬 "智能工作台的通用智能体，和应用的智能体，根本不是同一个 Agent。怎么能放在同一个页面？"

这个质疑，是从技术实现角度出发的——两个 Agent 是不同的实例、不同的配置、不同的运行逻辑，从工程视角看确实是两件事。
⚖
我的反驳

用户心智 ≠ 技术实现
把视角拉回用户身上

技术视角
Agent A、Agent B
是不同实例

不同配置、不同模型、不同运行逻辑——从工程角度看，这是两个独立的系统，自然不该放在同一个页面。

VS
用户视角
这就是一个
"干活的页面"

用户根本不会理解"工作台里有什么 Agent 概念"——他们只看到一个功能页面，处理"普通任务"和"应用任务"，是再自然不过的事。

  • 用户根本不会理解"智能工作台里有什么 Agent 概念"——这是一个工程内部的概念，不是用户认知的一部分
  • 用户会把智能工作台理解成一个功能页面——一个"干活的地方"
  • 在这个"干活的地方"，处理"普通任务"和"应用任务"，是再自然不过的事
  • Agent 之间的差异，是技术实现的细节，不应该外溢到用户认知层
🎯 设计师的关键立场
这是交互设计师作为"用户代言人"的关键时刻——
当组织内的讨论开始用技术语言主导产品决策时，
必须有人把视角拉回到用户身上。
C
✕ 强烈反对

大老板版本
左导航直接铺常用智能体

图位 04-C1
大老板提出的"左导航增加常用智能体"方案图
标注"一级导航被应用占用"为产品骨架被侵蚀的区域

在大老板的设想里，解决"应用没有家"这个问题的方法很直接——在最左侧一级导航中，直接把"常用智能体"铺出来。

我从三个维度强烈反对
维度 01 · 稳定性
骨架被侵蚀
一级导航是产品最稳定的"骨架"，承载的是产品核心架构。而应用是用户可以自由添加 / 删除的内容——把可变内容放进骨架，一级导航就不再稳定了。
维度 02 · 语义
属性错位
应用是「使用」属性（我用什么），一级导航承载的是「功能架构」语义（产品有什么）。两者属性根本错位。
维度 03 · 产品认知
"ADP 是什么"变模糊
一级导航被应用占满后，会让"ADP 是什么"这件事变得模糊——用户进来看不到平台的核心结构，只看到一堆别人配的应用。
D
✕ 未被采纳

与其做 C，不如拆 Tab
从根本定位上重新切分平台

⏱ 方案 C 和 D 几乎是同时被提出的，D 带有"反驳 C"的意味——但更重要的是，D 本身是一种更彻底的结构思路：不是为了应付应用入口的位置问题，而是从"搭建 vs 使用"这个根本定位上重新切分平台。
图位 04-D1
顶部 Tab 拆分方案图
高亮顶部双 Tab——「构建智能体」和「智能工作台」，标注"使用 / 搭建"分流

与其让一级导航被应用塞满变得不稳定，不如直接把"构建智能体"和"智能工作台"拆成最顶部的两个 Tab：

  • 「构建智能体」Tab：搭建 / 管理链路（应用开发、知识库、模型、Skills、平台管理……）
  • 「智能工作台」Tab：纯使用链路（对话、应用、定时任务、远程终端……）
这个方案的三重价值
价值 01 · 用户认知
真正的"工位"
智能工作台的左侧不再被理解成"通用 Agent 的历史对话列表"，而是真正"使用页面"的一级导航。整个工作台变成一个真正的"工位"，而不是一个混杂的对话窗口。
价值 02 · 商业价值
权限位天然清晰
员工权限位区分清晰——纯使用员工不需要看到管理页面，权限分配天然清晰。
价值 03 · 未来扩展
可独立售卖
若未来要面向纯使用场景的 B 端员工售卖（无智能体搭建场景的客户），结构上很方便整体拆出去单独售卖。
为什么这个方案最终也没被采纳——
大老板最终决策："构建 + 使用 一体，定位上不拆开。" 他坚持 ADP 就是一个完整的产品，从搭建到使用都在一个框架里。
E
✓ 最终落地

常用应用作为一级菜单
一个让所有人都能往前走的位置

图位 04-E1
最终落地方案图 — 「常用应用」作为一级菜单
标注"常用应用"一级菜单，对比此前方案的差异

在「智能工作台」之外，新增一个并列的一级菜单——「常用应用」。

  • 用户点击「常用应用」一级菜单后，可以直接和已构建好的应用进行对话
  • 应用列表在这个一级菜单的左侧栏内陈列，结构上保留了 B 方案中"类型→内容→任务"的层级思想
  • 既保留了产品的"使用沉淀"诉求 ✓
  • 又保持了"构建 + 使用"在同一产品框架下的整体性 ✓
"妥协方案" ≠ "失败方案"
它在产品的定位坚持、用户的使用诉求、组织内的协同共识之间，找到了一个 所有人都能往前走的位置。

而这条决策链上推演过的所有方案——A 的灵活、B 的统一抽象、C 的不稳定问题、D 的彻底拆分——没有一个是白做的。它们让最终方案 E 是被"选出来的"，而不是"被拍出来的"。

Section 06 / 其他功能

4.0 的其他迭代
一笔带过

围绕同一主题：让"使用"这件事，从"能用"走向"好用"，再走向"可沉淀、可流通、可经营"。

6.1 创建与配置类
支持对话式创建应用把"搭建"折叠进"使用"，用户用对话就能完成应用创建
